# 代码风格与约定（现状画像）

本文用于记录当前仓库的代码风格特点与一些隐含约定，便于后续统一格式、减少协作摩擦。样本主要来自 `src/` 与 `scripts/`。

## 总体风格

- **原型优先**：大量模块处于“骨架 + 占位实现”阶段，常见写法是 `throw new Error("Method not implemented.")` 或空方法体（例如 `src/cardgame/EventFlow.ts`、`src/cardgame/Zone.ts`）。
- **引擎模板混合自写代码**：`src/main.ts` / `src/LoadingUI.ts` 保留 Egret 模板的大段版权头与注释风格，而业务/原型代码更偏快速迭代（例如 `src/project/dule/DuleView.ts`）。
- **运行时驱动**：偏向在 UI/入口中直接拼装对象与流程（例如 `src/main.ts` 里完成资源加载、FairyGUI 包加载、View 切换）。

## 语言与模块组织

- **ESM**：`package.json` 设置 `"type": "module"`，源码普遍使用 `import/export`。
- **TypeScript 为主，脚本为 Node ESM**：`src/` 是 TS；`scripts/`（例如 `scripts/vite_manifest.js`）是 ESM 的 Node 脚本风格。
- **分层意图**：
  - 入口/生命周期：`src/main.ts`
  - 框架薄层：`src/framework/`
  - UI：`src/game/` + `src/project/`
  - 对战/规则原型：`src/cardgame/`、`src/hearthstone/`

## 命名与可读性习惯

- **类名/类型**：多数使用 PascalCase（例如 `Framework`、`ViewManager`、`EventFlow`）。
- **变量/方法**：多数使用 camelCase，但在“数据字段/动作参数”处出现 snake_case（例如 `src/hearthstone/actions/Draw.ts` 的 `source_zone_name`）。
- **拼写不一致会被保留下来**：例如 `ViewManager` 里曾出现 `context` 的拼写错误，会扩散到类型与调用侧（`src/framework/managers/ViewManager.ts`）。
- **中英混排**：注释、日志与文档大量中文；标识符以英文为主。

## 类型与接口风格（TS）

- **类型标注呈两极**：
  - 框架/核心结构倾向“先把类型形状写出来”（例如 `src/cardgame/EventFlow.ts` 的 `GameEvent`、`TriggerHandle`）。
  - 业务原型倾向“先跑起来再补类型”，经常省略参数/成员类型（例如 `src/hearthstone/actions/Draw.ts`）。
- **`any`/弱类型常见**：例如 `IView.context: any`、`ViewManager.context: any`（`src/framework/managers/ViewManager.ts`）。
- **占位 API**：空实现或抛错的 API 仍然被上层引用，导致 `tsc` 会直接暴露“接口不一致”问题（例如 `DuleView` 调用 `battle.addFlowHandler`，但 `Battle` 并未实现对应方法）。

## 代码格式与细节习惯（现状）

- **缩进不完全统一**：多数 TS 代码是 4 空格缩进，但也存在 Tab（例如 `scripts/vite_manifest.js` 的 JSON 缩进参数使用了 `"\t"`）。
- **分号不统一**：有的文件是“全分号”（偏模板），有的文件是“几乎不写分号”（偏原型）。
- **引号风格**：多用双引号；少量模板/外部代码保留原始写法。
- **`let` 使用偏多**：即使不重新赋值也常用 `let`（例如 `src/main.ts`、`src/project/dule/DuleView.ts`）。
- **快速调试输出**：大量 `console.log(...)` 直接写在入口/回调中。

## 与当前仓库约定的差异（需要注意）

仓库约定（见根目录 `AGENTS.md`）强调：

- TypeScript 目标 ES2020
- **四空格缩进**
- **保留分号**

但现状中存在“新写文件未严格遵守分号/缩进”的情况，后续如果要统一风格，建议先明确“以约定为准”还是“以现状为准”，避免反复格式抖动。

## 建议的最小统一口径（可选）

如果希望后续减少 diff 噪音、降低维护成本，可以先落实以下“最小集合”：

- TS：统一 **4 空格缩进 + 分号**（与 `AGENTS.md` 对齐）。
- 命名：业务参数字段尽量用 `camelCase`；仅在“序列化数据结构”里保留 `snake_case`。
- 导入：统一不写 `.ts` 扩展名（例如把 `import { DuleView } from "./project/dule/DuleView.ts";` 改为无扩展名）。
- 修正明显拼写错误并一次性全局替换（例如 `context` 相关字段/路径），避免继续扩散。
