# 对战流程（抽牌→出牌→效果→攻击→结束回合）

本文把“玩家从抽牌到打出一张牌、发动效果与攻击、直到结束回合”的整套做法，落到本项目的目标架构：`ActorOperate + Turn 状态机 + EventFlow（触发链/回滚）+ Table/Zone 顺序`（见 `CARDGAME.MD`）。

## 总原则：一切交互先变成 Event

- UI 不直接改数据（Zone/Card/血量/资源），而是发起 `ActorOperate`。
- `ActorOperate` 只负责 **校验** + **生成事件**（Event）入队。
- `EventFlow` 负责 **取事件→按优先级找触发器→产出 Action→记录变更→派生新事件**。
- 任何“效果”“被动”“奥秘/陷阱”“死亡结算”“连锁触发”都统一走事件管线，避免散落 if-else。
- 回滚：每个事件对应一帧 `EventStackFrame`（事件 + 执行动作 + 变更记录），需要重算/AI 预测时可回退到任意帧。

## 核心对象分工（建议口径）

- `Battle`
  - 组织多轮 `Turn`、胜负判定、当前行动方切换。
  - 持有：`BattleTable`、`EventFlow`、（当前）`Turn`。
- `Turn` / `TurnStateMachine`
  - 定义阶段（例如 `ready -> draw -> main -> end`），阶段切换时生成阶段事件。
- `BattleTable`
  - 纯数据容器：Actor/Player、各 `Zone`、卡牌实例索引（CardInstance Registry）。
  - 处理“跨 Zone 移动”带来的顺序同步（让 EventFlow 触发排序与 Zone 顺序一致）。
- `Zone`
  - 管理卡牌列表、顺序与容量规则；卡牌移动/顺序改变时，通知 EventFlow 更新相关触发器排序键。
- `CardTemplate` / `CardInstance` / Components
  - Template：静态数据（费用、描述、组件蓝图）。
  - Instance：运行时实体（攻击/生命/状态/所在 Zone/所属者）。
  - Component：只包含 **数据 + 触发器声明 + 动作生成**，并通过 EventFlow 响应事件。
- `ActorOperate`
  - 描述“某 Actor 在某 Turn/Phase 下可执行的操作集合”，用于 UI 获取“可操作列表”和做前置校验。

## 回合主流程（从开始到结束）

下面以“炉石风格”阶段举例，事件命名仅作约定建议。

### 1) 进入回合（TurnStart）

1. `Turn.start()` 设置当前阶段为 `ready`
2. 入队事件：`TURN_START { actorId }` / 或 `PHASE_ENTER { phase:'ready' }`
3. 触发器可在此处理：回合开始刷新、状态衰减/解除、资源准备等

### 2) 抽牌阶段（Draw）

1. 入队：`PHASE_ENTER { phase:'draw' }`
2. 系统自动抽牌：入队 `DRAW_REQUEST { actorId, count: 1 }`
3. `EventFlow` 结算 `DRAW_REQUEST`（推荐拆成可回滚 Action）：
   - `ZonePop`：从 `deck` 弹出 1 张（若空：派生 `FATIGUE_DAMAGE`/`LOSE_GAME`）
   - `ZonePush`：若手牌未满则推入 `hand`，否则派生 `BURN_CARD`（爆牌）
   - 派生语义事件（用于触发监听）：`CARD_DRAWN`、`CARD_ENTER_ZONE`

### 3) 主阶段（Main：出牌/技能/攻击）

进入 `PHASE_ENTER { phase:'main' }` 后，UI 允许玩家发起 `ActorOperate`：

- `OP_PLAY_CARD`：出牌（手牌→场上/墓地等）
- `OP_USE_SKILL`：用技能
- `OP_DECLARE_ATTACK`：宣告攻击
- `OP_END_TURN`：结束回合

Operate 的校验通常包括：阶段、归属、资源（费用/水晶）、目标合法性（嘲讽/潜行）、状态限制（冻结/沉默）等。

## 出牌流程（推荐事件链）

出牌建议分“请求事件→规则/支付→移动→结算→收尾”，方便插入/打断/回滚。

1. `PLAY_CARD_REQUEST { playerId, cardInstanceId, targets, options }`
2. 规则层触发后派生：
   - `PAY_COST { playerId, cost }`
   - `MOVE_CARD { cardInstanceId, from:'hand', to:'battlefield'|'graveyard'... }`
   - `CARD_PLAYED { cardInstanceId }`（语义事件：给“每当你打出一张牌后”监听）
3. 分类型结算：
   - 法术：`RESOLVE_SPELL { cardInstanceId, targets }` → 派生 `DEAL_DAMAGE`/`SUMMON`/`DRAW_REQUEST`… → `MOVE_CARD` 到墓地
   - 随从：`SUMMON_MINION { cardInstanceId, position }` → `BATTLECRY`/`ON_SUMMON` 触发链
4. 收尾：`AFTER_PLAY_CARD { playerId, cardInstanceId }`

触发器建议监听“语义事件”（例如 `CARD_PLAYED`、`SUMMON_MINION`、`RESOLVE_SPELL`），而不是监听 UI 行为，避免耦合。

## 攻击流程（推荐事件链）

攻击拆成“宣告→伤害→死亡结算→收尾”，形成稳定的结算窗口。

1. `DECLARE_ATTACK { attackerId, defenderId }`
2. 校验并派生：`ATTACK_START { attackerId, defenderId }`
3. 派生：`DEAL_DAMAGE { source: attackerId, target: defenderId, amount }`
4. 若互殴，再派生一条：`DEAL_DAMAGE { source: defenderId, target: attackerId, amount }`
5. `DEAL_DAMAGE` 结算阶段建议顺序：
   - 先触发护盾/免疫/减伤等（触发器可改写 amount 或转化为 0）
   - 再应用伤害（Action 记录 hp/armor 变更）
   - 派生：`DAMAGE_DEALT`、`DAMAGE_TAKEN`
6. 统一死亡检查：`CHECK_DEATHS` → 对每个死亡单位派生 `UNIT_DIES`（亡语/复生等在此链路继续派生事件）
7. 收尾：`ATTACK_END`（标记已攻击、消耗攻击次数等）

## 结束回合流程（End）

1. 玩家点击结束回合：`END_TURN_REQUEST { playerId }`
2. 派生：`PHASE_EXIT { phase:'main' }` → `TURN_END { playerId }`
3. 在 `TURN_END` 链路做清理：
   - 到期 Buff/状态结算
   - 重置“本回合已攻/已用”标记
   - “回合结束时”触发器执行
4. Battle 切换行动方，进入对手 `TURN_START`

## UI 与逻辑解耦建议

- UI 层只做：
  - 展示当前 `BattleTable` 的状态（Zone、CardInstance、资源、血量）
  - 收集输入并发起 `ActorOperate`
  - 根据 EventFlow 的变更记录播放动画（抽牌移动、扣血飘字、死亡移除）
- 逻辑层只做：
  - 计算合法操作（`Actor.listOperates(context)`）
  - `EventFlow` 结算并产出变更记录（用于 UI 驱动表现）

## 与当前代码骨架的落点（供实现时对齐）

- `src/cardgame/EventFlow.ts`：实现队列/栈/触发器索引 + `resolveNext()`/`rollbackTo()`
- `src/cardgame/Turn.ts`：实现 `start/advance/end` 并在阶段切换时入队 `PHASE_ENTER/EXIT`
- `src/cardgame/Actor.ts`：实现 `listOperates(context)`（把“阶段限制/资源限制/目标合法”集中到这里）
- `src/hearthstone/actions/*`：作为 Action/指令集的实现载体（可回滚），由 EventFlow 调用
